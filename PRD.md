# DataHelp 产品需求文档（复盘补充版）

> 文档版本：v1.0  
> 补充日期：2026-06-21  
> 产品阶段：可运行原型，持续迭代中  
> 文档说明：本文件基于已完成的原型、测试记录与用户访谈回填，用于统一产品目标、功能边界与验收方式；并非项目启动时已存在的历史 PRD。

## 1. 产品背景

商科学生在处理 CSV、Excel 等业务数据时，常见困难并非单纯缺少分析工具，而是缺少完成任务的下一步指引：

- 不知道数据字段分别代表什么，也不知道哪些字段可以回答业务问题；
- 不知道从数据清洗、基础指标、分组比较、趋势分析中先做哪一步；
- 即使得到图表或结论，也不理解分析方法、指标含义与结论边界；
- 作业或汇报完成后，缺少可回看、可复用的学习材料。

上述问题来自项目发起者及其周围商科同学的实际学习体验，并通过与同学、师兄师姐及老师的非正式交流得到补充验证。该调研为探索性访谈，未形成具有统计代表性的样本量结论。

## 2. 产品定位

DataHelp 是一个面向商科初学者的本地数据分析学习 Agent。

它不只输出分析结论，也要让用户理解“为什么这样分析、这个指标意味着什么、结论不能说明什么、下次遇到类似数据可以如何继续”。

产品不定位为专业统计软件、机器学习平台或通用 BI 工具，而是降低非技术用户完成基础业务数据分析任务的门槛。

## 3. 目标用户与典型场景

### 3.1 目标用户

- 工商管理、市场营销、运营、管理学等专业的学生；
- 初次接触业务数据分析、会使用 Excel 但缺少系统方法的用户；
- 需要完成零售、营销、用户、门店、销售等数据分析作业或汇报的用户。

### 3.2 典型场景

1. 用户拿到一份 CSV 或 Excel 数据集，不知道从哪些字段和问题开始分析。
2. 用户需要完成数据分析作业，希望同时获得分析结果和可学习的方法解释。
3. 用户需要将分析结果整理为可回看的 PDF、可检查的 Excel 和可演示的 HTML 页面。

## 4. 核心问题与产品目标

### 4.1 核心问题

用户面对数据时，常陷入“不会提出问题—不会清洗数据—不会解释结果”的连续断点。

### 4.2 产品目标

1. 帮助用户快速理解数据集的字段、质量状况和可分析方向。
2. 以可解释的分析流程完成基础业务分析，而非只给出一段结论。
3. 将分析过程沉淀为可回看、可展示、可继续探索的交付物。
4. 对数据不足或不适合直接执行的高级分析明确说明边界，不强行得出结论。

### 4.3 非目标

- 不承诺替代专业数据分析师或统计软件。
- 不在缺少字段、样本量不足或用户未明确目标时强制执行预测、因果推断等高级方法。
- 不把模型生成的结论视为无需人工检查的业务事实。

## 5. 核心用户路径

```text
导入 CSV / Excel 数据
  → 识别数据规模、字段与潜在业务含义
  → 数据质量检查
  → 基础指标、分组、排名、趋势等多维分析
  → 提炼发现、业务建议、边界与进阶方向
  → 输出教学型报告及 Excel / HTML / PDF 交付物
```

## 6. 功能需求

### 6.1 数据导入与预处理

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| FR-01 | 支持 CSV、XLS、XLSX 文件输入。 | 对受支持格式给出明确的成功或失败提示。 |
| FR-02 | 识别字段名、字段类型、数值/分类/日期字段及基础数据规模。 | 报告中能说明可用于分析的主要字段及其业务含义假设。 |
| FR-03 | 检查缺失值、重复值、异常值及常见业务数据风险。 | 对发现的风险说明影响范围与处理建议，而非直接静默删除。 |

### 6.2 分析与教学输出

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| FR-04 | 默认完成基础指标、分组排名、趋势与数据质量等基础分析。 | 可针对数据条件输出对应结果；缺少日期等必要字段时须说明原因。 |
| FR-05 | 每个核心发现包含结果、方法、指标/公式、字段来源、业务含义、风险边界与复用方向。 | 不仅有“发现—建议”，还包含可学习的解释。 |
| FR-06 | 对高级方法采用分层策略：满足条件时推荐，只有在用户明确需要且数据条件满足时执行。 | 报告中说明推荐原因、所需字段及不能回答的问题。 |
| FR-07 | 在 `beginner_summary` 模式下强制输出 12 节教学结构。 | 不遗漏数据概览、字段识别、质量检查、基础分析、发现、建议、边界与教学总结等核心内容。 |

### 6.3 交付物

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| FR-08 | 输出 PDF 学习笔记，用于回看本次分析摘要。 | PDF 包含本次分析的核心结论与教学说明。 |
| FR-09 | 输出 Excel 结果文件，提供源数据/统计或透视分析及可读解释。 | 不仅生成空模板，需包含本次分析相关内容。 |
| FR-10 | 输出 HTML 演示页面，用于结果展示和分享。 | 页面包含本次分析结论，可作为汇报材料的基础。 |

### 6.4 运行模式与稳定性

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| FR-11 | 支持 `beginner_summary`、`standard_report`、`audit_report` 三种输出模式。 | 不同模式在教学解释深度与审计详细度上可区分。 |
| FR-12 | 支持在同一任务内先多维分析，再根据结果继续深挖。 | 不依赖“一次一个细粒度工具调用”完成全部分析。 |
| FR-13 | 将完整 Business Analysis Skill 作为运行时指令的一部分。 | 教学规则不因上下文裁剪而只剩局部摘要。 |

## 7. 关键产品与技术决策

### 7.1 为什么不沿用普通 Coding Agent 的线性流程

Pico 等轻量级 Coding Agent 的典型流程适合“读文件—改代码—运行—查看报错—继续修复”。但数据分析需要先发散地观察字段、质量、指标与多维切片，再收敛为关键发现和建议。

因此 DataHelp 将核心分析方式调整为：先获取数据概览，再生成可一次完成多维分析的 pandas 脚本，运行后根据结果继续深挖，最后编译为固定的教学报告结构。

### 7.2 教学优先，而非只追求输出长度

对初学者而言，输出的价值不只在于结论数量，而在于理解分析依据。产品通过固定章节、方法解释、指标解释、风险边界和下一步建议，让一次分析变成可复用的学习过程。

### 7.3 交付物应与分析结论绑定

PDF、Excel、HTML 不应只是独立模板。三类交付物需要承接 Agent 本次输出的分析结论，避免出现“对话里有结论、文件里只有空壳”的断裂体验。

## 8. 原型测试与 Bad Case 复盘

### 8.1 测试方式

使用公开业务/电商数据集进行原型测试。数据集并非固定唯一来源，产品目标是支持结构相对清晰的公开 CSV 或 Excel 业务数据；架构审计中曾使用 Olist Brazilian E-commerce 数据集进行对比测试。

### 8.2 已发现问题

- 初版输出只有约 5–6 节，部分关键分析步骤和教学要素缺失；
- 初版报告约 4,600 字符，内容完整度不足；
- 细粒度工具逐个调用，难以在有限步骤内完成多维分析；
- Skill 规则与上下文预算不足，导致模型无法获得完整教学要求；
- 初版交付物与 Agent 最终分析结论脱节。

### 8.3 改进方向

- 用脚本化多维分析替代逐个细粒度工具调用；
- 增加上下文和工具输出预算，保证完整规则可用；
- 采用 12 节固定教学报告结构；
- 将分析结论写入三类交付物；
- 继续以失败案例驱动规则、流程和输出稳定性的迭代。

### 8.4 当前结果的正确表述

在同类测试任务中，重构后报告输出规模由约 4,600 字符增加到 15,200+ 字符，章节由约 5–6 节扩展至 12+ 节。该结果用于说明报告完整度和教学覆盖度的提升，**不等同于系统响应速度或性能提升 3 倍**。

## 9. 后续迭代方向

1. 建立更多公开数据集与失败案例的回归测试集。
2. 持续收集真实用户对教学解释、交付物可用性和分析建议的反馈。
3. 明确不同输出模式的用户预期与适用场景。
4. 优化多表关联、数据异常处理和分析结果稳定性。
5. 为高级分析建立更明确的前置条件和人工确认机制。

## 10. 面试使用说明

本 PRD 是在原型形成后对产品思考、功能边界和测试复盘的系统化沉淀。面试中应如实表达为：

> 在原型迭代后，我将前期零散的需求判断、Skill 规则、测试发现和功能边界整理为 PRD，用于统一后续迭代与验收，而不是把它说成项目从第一天就已完成的文档。
